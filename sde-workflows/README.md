# Data Extraction Service

A serverless pipeline that extracts filtered subsets of Apache Iceberg tables (via **PyIceberg + PyArrow**) and loads them into **SQL Server Express** on Fargate, generating `.bak`/`.mdf` artifacts for tenant delivery.

## Features

- **2-step workflow** — Step 1 generates a reviewable mapping config; Step 2 ingests using that config. Or use `--auto` for one-shot.
- **Multi-table support** — map and ingest all tables in a namespace with `--all-tables`, or specify multiple `--table` args.
- **Full load** — reads the entire Iceberg table and replaces (truncate + insert) the SQL Server table.
- **Incremental load** — tracks the last-ingested Iceberg snapshot and only loads new data on subsequent runs.
- **Automatic column mapping** — intelligently maps Iceberg columns to SQL Server columns even when names differ (see below).
- **Reviewable mapping config** — auto-generated `mapping_config.json` lets you review, correct, and reuse column mappings.
- **Auto DDL** — automatically creates the target SQL Server table based on the mapping config if it doesn't exist.
- **Configurable via `.env`** — all connection details and settings in one file.
- **CLI overrides** — override table names, load mode, and batch size from the command line.
- **Optional high-speed writes with BCP** — set `WRITE_BACKEND=bcp` for much faster SQL Server bulk loads on large datasets.

## Automatic Column Mapping

When the SQL Server target table **already exists** with different column names, the script automatically matches source → target columns using a multi-layer strategy:

| Priority | Strategy | Example |
|---|---|---|
| 1 | **Exact match** | `order_id` → `order_id` |
| 2 | **Case-insensitive** | `order_id` → `Order_Id` |
| 3 | **Normalized** (strip `_`, `-`, spaces) | `order_id` → `OrderId` |
| 4 | **Abbreviation expansion** | `cust_name` → `CustomerName` |
| 5 | **Fuzzy match** (similarity ≥ 70%) | `ordr_id` → `OrderID` |
| 6 | **Type compatibility** tiebreaker | Boosts confidence for same-type pairs |

When the target table **does not exist**, it is auto-created using a naming convention transform (default: `PascalCase`). Override with `--naming-convention` or `NAMING_CONVENTION` in `.env`.

Each run logs the mapping decisions with confidence scores so you can verify:

```
  SOURCE                          →  TARGET                          STRATEGY         100%
  ------------------------------------------------------------------------------------------
  order_id                        →  OrderID                         normalized       95%
  cust_name                       →  CustomerName                    abbreviation     90%
  order_dt                        →  OrderDate                       abbreviation     90%
  amt                             →  TotalAmount                     fuzzy+type       78%
```

## Prerequisites

- Python 3.10+
- Access to an Iceberg catalog (REST, Hive, Glue, etc.)
- SQL Server with an ODBC driver installed ([Microsoft ODBC Driver 18 for SQL Server](https://learn.microsoft.com/en-us/sql/connect/odbc/download-odbc-driver-for-sql-server))

## Quick Start

### 1. Install dependencies

```bash
pip install -r requirements.txt
```

### 2. Configure

Copy the example env file and fill in your values:

```bash
cp .env.example .env
```

Edit `.env` with your Iceberg catalog and SQL Server connection details.

### 3. Run

The script uses a **2-step workflow**:

```bash
# ━━━ STEP 1: Generate mapping config (NO data is moved) ━━━

# Single table
python ingest.py --table default.orders

# Multiple tables
python ingest.py --table default.orders --table default.customers

# All tables in the Iceberg namespace
python ingest.py --all-tables
```

This creates `mapping_config.json` with auto-detected column mappings.
Open it, review/edit the mappings, then proceed to Step 2.

```bash
# ━━━ STEP 2: Run extraction using the mapping config ━━━

# Run ALL tables in the mapping config
python ingest.py --run

# Run a specific table
python ingest.py --run --table default.orders

# Run with incremental mode override
python ingest.py --run --mode incremental

# Use a custom mapping file
python ingest.py --run --mapping my_mappings.json
```

```bash
# ━━━ ONE-SHOT: Generate + ingest immediately (skip review) ━━━

python ingest.py --run --auto --table default.orders
python ingest.py --run --auto --all-tables
```

### Mapping Config (`mapping_config.json`)

Generated by Step 1, consumed by Step 2. Keyed by `{namespace}_{table}`:

```json
{
  "default_orders": {
    "source": { "namespace": "default", "table": "orders" },
    "target": { "schema": "dbo", "table": "Orders" },
    "generated_at": "2026-02-17T10:30:00+00:00",
    "naming_convention": "PascalCase",
    "create_table_if_missing": true,
    "load_mode": "full",
    "columns": [
      {"source": "order_id",  "target": "OrderId",       "source_type": "int64",  "target_type": "BIGINT",        "strategy": "naming-convention", "confidence": 1.0,  "include": true},
      {"source": "cust_name", "target": "CustomerName",   "source_type": "string", "target_type": "NVARCHAR(MAX)", "strategy": "abbreviation",     "confidence": 0.90, "include": true},
      {"source": "_internal", "target": null,              "source_type": "string", "target_type": null,            "strategy": "unmatched",        "confidence": 0.0,  "include": false}
    ]
  }
}
```

You can edit:
- `"target"` — rename target columns
- `"target_type"` — override SQL Server types (e.g. `DECIMAL(18,4)` instead of `FLOAT`)
- `"include": false` — skip columns you don't want
- `"include": true` on an unmatched column + fill in `"target"` — manually map it

## Configuration Reference

All settings are read from the `.env` file. See `.env.example` for the full list.

### Iceberg Settings

| Variable | Description | Default |
|---|---|---|
| `ICEBERG_CATALOG_TYPE` | Catalog type: `rest`, `hive`, `glue`, `sql`, `dynamodb` | `rest` |
| `ICEBERG_CATALOG_NAME` | Logical catalog name | `my_catalog` |
| `ICEBERG_CATALOG_URI` | REST catalog endpoint | — |
| `ICEBERG_S3_ENDPOINT` | S3-compatible storage endpoint | — |
| `ICEBERG_S3_ACCESS_KEY` | S3 access key | — |
| `ICEBERG_S3_SECRET_KEY` | S3 secret key | — |
| `ICEBERG_WAREHOUSE` | Warehouse path (e.g. `s3://bucket`) | — |
| `ICEBERG_NAMESPACE` | Iceberg namespace / database | `default` |
| `ICEBERG_TABLE` | Table name to ingest | — |

### SQL Server Settings

| Variable | Description | Default |
|---|---|---|
| `SQLSERVER_HOST` | Server hostname or IP | `localhost` |
| `SQLSERVER_PORT` | Port | `1433` |
| `SQLSERVER_DATABASE` | Target database | — |
| `SQLSERVER_SCHEMA` | Target schema | `dbo` |
| `SQLSERVER_TABLE` | Target table name | (same as source) |
| `SQLSERVER_USERNAME` | Login username | — |
| `SQLSERVER_PASSWORD` | Login password | — |
| `SQLSERVER_AUTH_TYPE` | `sql_login`, `windows`, or `azure_ad` | `sql_login` |
| `SQLSERVER_DRIVER` | ODBC driver name | `ODBC Driver 18 for SQL Server` |
| `SQLSERVER_TRUST_CERT` | Trust self-signed certs | `yes` |

### Extraction Settings

| Variable | Description | Default |
|---|---|---|
| `LOAD_MODE` | `full` or `incremental` | `full` |
| `BATCH_SIZE` | Rows per INSERT batch | `5000` |
| `WRITE_BACKEND` | SQL write backend: `pyodbc` or `bcp` | `pyodbc` |
| `BCP_PATH` | Path to `bcp` executable | `bcp` |
| `BCP_BATCH_SIZE` | bcp commit batch size (`-b`) | `50000` |
| `BCP_TEMP_DIR` | Temp directory for generated bcp files | system temp |
| `BCP_FIELD_TERMINATOR` | Delimiter used in bcp input files | `|~|` |
| `BCP_ROW_TERMINATOR` | Row delimiter used in bcp input files | `~|~\n` |
| `STATE_FILE` | Path to snapshot state file | `.ingestion_state.json` |
| `LOG_LEVEL` | Python log level | `INFO` |
| `NAMING_CONVENTION` | Target column naming: `PascalCase`, `camelCase`, `UPPER_SNAKE`, `lower_snake`, `as_is` | `PascalCase` |
| `MAPPING_FILE` | Path to mapping config file | `mapping_config.json` |

### High-speed SQL writes with BCP (optional)

For larger tables, you can switch SQL writes from `pyodbc` to SQL Server's native bulk copy utility:

```bash
# .env
WRITE_BACKEND=bcp
BCP_PATH=bcp
```

Notes:
- If `bcp` is unavailable or the data contains unsupported types/delimiter collisions, the script automatically falls back to the `pyodbc` path.
- Ensure SQL Server command line tools are installed wherever the script runs.

## How It Works

1. **Step 1 — Generate mapping** — Connects to both Iceberg and SQL Server, reads schemas, auto-matches columns using 6-layer matching (exact → fuzzy), and writes `mapping_config.json`. No data moves.
2. **Review** — User opens the JSON, verifies/edits column mappings, types, and include flags.
3. **Step 2 — Ingest** — Reads the (possibly edited) mapping config, creates target tables if needed, reads Iceberg data into Arrow, and bulk-inserts only the included & matched columns into SQL Server.
4. **Track** — Saves the current Iceberg snapshot ID so the next incremental run picks up where it left off.
